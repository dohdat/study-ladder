# Project Agent Instructions

- For source edits, keep iteration fast. Run `npm.cmd run typecheck` only when changing TypeScript contracts, state shape, build config, or logic where a static check can catch real risk. Run a focused `npm.cmd run test:unit -- <test-file>` when touching tested logic. Do not run the full production build after every small edit while still iterating.
- Chrome should load the exported unpacked folder: `D:\Leetcode + System Design extension\dist-unpacked`. Do not load the repo root in Chrome; the old repo-root/fixed-key extension ID was blocked/stale. The current normal Chrome ID may change, so let the reload/native-host scripts detect it from Chrome preferences instead of hardcoding it.
- After source changes that need to show up in the Chrome-loaded extension, run `npm.cmd run export:extension` so `dist-unpacked` is rebuilt, then run `npm.cmd run reload:extension`. `npm.cmd run export:extension` performs the full Vite build and copies the runtime files into `dist-unpacked`.
- For quick visual/UI iteration only when the active Chrome extension is loaded from the repo root, prefer `scripts\dev-extension.cmd` over `npm.cmd run dev:extension` to avoid npm startup overhead. It runs a quick Vite build into `out/`, refreshes the stable extension pages under `pages/`, reloads the unpacked extension, and reopens `chrome-extension://.../pages/index.html`. Do not use this as the normal persisted Chrome workflow while Chrome is loaded from `dist-unpacked`, because it does not update `dist-unpacked`.
- For repeated visual tweaking, use `scripts\watch-extension.cmd` only with a repo-root-loaded dev extension. It keeps Vite warm, rebuilds `out/` on file changes, and reloads the extension after each rebuild, but it does not export to `dist-unpacked`.
- For screenshot-level visual tweaks, run `scripts\dev-extension.cmd` when the user needs to see the change in Chrome. Skip `npm.cmd run typecheck` for CSS/layout-only edits unless the touched code or compiler surface is risky.
- Before handing work back for a final packaged change, or before commit/push/release work, run one final `npm.cmd run export:extension` and `npm.cmd run reload:extension`. The persisted unpacked extension loads `dist-unpacked`, so Chrome can show stale UI unless that folder is exported and reloaded.
- If the reload helper cannot find the extension ID, pass `-- -ExtensionUrl "<current chrome-extension://.../pages/index.html>"`, pass `-- -ExtensionId <id>`, or set `STUDY_LADDER_EXTENSION_ID`.
- If native host access stops working after Chrome assigns a new extension ID, run `npm.cmd run install:native-host` after loading `dist-unpacked`.
- For docs-only or investigation-only changes, skip build/reload unless the user asks to verify the extension bundle.
- Timing note: the project now uses Vite instead of Next for extension builds. `npm.cmd run export:extension` is the normal persisted-extension build because Chrome is loaded from `dist-unpacked`; `scripts\dev-extension.cmd` remains the fast repo-root-only visual loop. Focused map tests are usually under 1 second of test runtime.
